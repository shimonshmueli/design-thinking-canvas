2.6 Briefing (the Design Brief)简报撰写（设计简报）

What it is

Writing the Problem Definition itself — the document that closes the first diamond. A good brief states the problem, the users and stakeholders, their needs/wants/aspirations, the chosen solution point-of-view, objectives and constraints, and measurable success criteria. It frames the solution space without prescribing the solution.撰写问题定义本身——这份文档标志着第一个钻石的收尾。一份好的简报会说明问题、用户与利益相关者、他们的需求/诉求/愿景、所选定的解决方案视角、目标与限制条件，以及可衡量的成功标准。它为解决方案空间划定了框架，但并不预先规定具体的解决方案。

How to use it

  1. Write the problem statement as a need, not a feature ('nurses lose track of X during shift changes', not 'build an app for X').把问题陈述写成一种『需求』，而不是『功能』（应写成『护士在交班时会遗漏 X 的追踪』，而不是『为 X 开发一款应用』）。
  2. Make success criteria observable and measurable.让成功标准可观察、可衡量。
  3. State constraints honestly — budget, technology, regulation, time.诚实地列出限制条件——预算、技术、法规、时间。
  4. Review with stakeholders and get explicit commitment; the brief is a contract for the second diamond.与利益相关者一起评审，并取得明确的承诺；简报就是第二个钻石的『合约』。

Watch out for

A brief that smuggles in the solution ('the problem is that we don't have a mobile app') closes the solution space before Ideate even opens it.一份夹带了解决方案的简报（例如『问题在于我们没有一款移动应用』），会在构思阶段还没开始之前，就提前关闭了解决方案空间。
