Briefing (the Design Brief)

What it is

Writing the Problem Definition itself — the document that closes the first diamond. A good brief states the problem, the users and stakeholders, their needs/wants/aspirations, the chosen solution point-of-view, objectives and constraints, and measurable success criteria. It frames the solution space without prescribing the solution.

How to use it

  1. Write the problem statement as a need, not a feature ('nurses lose track of X during shift changes', not 'build an app for X').
  2. Make success criteria observable and measurable.
  3. State constraints honestly — budget, technology, regulation, time.
  4. Review with stakeholders and get explicit commitment; the brief is a contract for the second diamond.

Watch out for

A brief that smuggles in the solution ('the problem is that we don't have a mobile app') closes the solution space before Ideate even opens it.
