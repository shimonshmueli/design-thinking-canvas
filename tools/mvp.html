4.2 Minimum Viable Product (MVP)最小可行产品（MVP）

↓ Jump to workspace

What it is

The smallest version of a solution that lets you test its core value hypothesis with real users at real stakes. An MVP is a learning instrument, not a small product: it exists to answer 'do people actually want and use this?' with evidence, before major investment.解决方案的最小版本，让你能够以真实的风险，向真实用户检验其核心价值假设。MVP 是一种学习工具，而不是一个『缩小版的产品』：它存在的目的，是在投入重大资源之前，用证据回答『人们是否真的想要并会使用它？』这个问题。

How to use it

  1. State the riskiest assumption in the value proposition.明确指出价值主张中风险最高的那个假设。
  2. Design the smallest thing that tests exactly that assumption — it may be a concierge service, a landing page, or a single-feature build.设计一个能精确检验这个假设的最小事物——它可以是人工代劳服务、一个落地页，或是只有单一功能的构建版本。
  3. Define success metrics before launch.在上线之前先定义好成功指标。
  4. Ship to real users, measure, and decide: persevere, pivot, or kill.投放给真实用户、进行测量，然后做出决定：坚持、转向，还是放弃。

Watch out for

The 'small first release' misreading — cramming a full roadmap into v1 and calling it an MVP. If it doesn't test a named assumption, it's not one.警惕把 MVP 误解为『小规模的首个发布版本』——把完整路线图硬塞进 v1 版本，再称之为 MVP。如果它没有在检验某个明确指出的假设，那它就不是真正的 MVP。
